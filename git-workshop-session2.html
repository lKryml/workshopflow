§1 — setup & configuration

Configure Git — Your Identity

Every commit you make is stamped with your name and email. Set this once globally and it applies to all repos on this machine.

  • Use your real name — it appears in team history and on GitHub Professional projects, employer reviews, open source
  • Use the same email as your GitHub account So contributions are linked to your profile activity graph
  • init.defaultBranch main avoids the old "master" name Industry standard since 2020
Terminal / Git Bash
$ git --version git version 2.43.0 $ git config --global user.name "Your Full Name" $ git config --global user.email "you@email.com" $ git config --global init.defaultBranch main $ git config --list # verify all settings
§2 — starting a project

Initializing a Repository

Running git init creates a hidden .git/ folder. That folder is the repository — it stores every version of your files.

📁 my-first-repo/
├── README.md
└── .git/ (hidden — Git lives here)
├── HEAD · config · objects/ · refs/
Terminal
$ mkdir my-first-repo && cd my-first-repo $ git init Initialized empty Git repository in .../my-first-repo/.git/ # Create README and make first commit $ echo "# My First Repo" > README.md $ git status # shows README untracked (red) $ git add README.md $ git commit -m "Initial commit: add README" $ git log --oneline 9a8b7c6 Initial commit: add README

§3 — reading history

Reading git log

Every commit has a unique hash (like a receipt number), the author, date, and message. Use --oneline for a compact view.

  • Hash — 2a3e4f5 — unique 7-char IDUse first 7 chars to reference any commit
  • Message — describes what changed & whyWrite for your future self (and teammates)
  • git log --oneline --graph --allVisualize branch topology
git log --oneline --graph
$ git log --oneline --graph * 2a3e4f5 Add paragraph to index.html (2 min ago) * 1b2c3d4 Add index.html with greeting (5 min ago) * 9a8b7c6 Initial commit: add README (10 min ago) # Full log shows author + date: $ git log # press q to exit # Jump to any point in history: $ git show 9a8b7c6 # inspect a commit
§4 — branching

What is a Branch?

A branch is a lightweight pointer to a commit. Creating one is instant — no files are copied. Branches isolate your work from main.

  • Independent line of workChanges on a branch don't affect main or other branches
  • Free to create & deleteBranches are just labels — no storage cost
  • Merge when readyBring changes back into main once reviewed & tested
Commands
# Create a branch $ git branch feature-nav # Switch to it $ git checkout feature-nav $ git switch feature-nav # modern # Create + switch in one step $ git checkout -b feature-nav # List all branches (* = current) $ git branch * main feature-nav # Merge branch into current $ git checkout main $ git merge feature-nav
Visual timeline
Branch off to work independently: main ●── ●── ● \ feature ●── ●── ● Merge (fast-forward, no diverge): main ●── ●── ●── ●── ●── ● Merge (diverged — creates merge commit): main ●── ●── ●────────── M \── ●── ●──/ feature M = merge commit (two parents)
§5 — team simulation

Merge Conflicts — The Reality

When two branches change the same lines in the same file, Git can't decide which to keep — it stops and asks you to decide. This is normal. It's not a bug, it's a feature.

Alice — branch: alice
team.txt line 2:
"Alice is working on the login feature."
Bob — branch: bob
team.txt line 2:
"Bob is working on the payment gateway."
After: git merge bob
CONFLICT (content): Merge conflict in team.txt Automatic merge failed; fix conflicts and then commit the result. $ git status # shows "both modified: team.txt"
§5 — reading conflicts

Understanding Conflict Markers

Git inserts markers directly into the file. You must manually choose what to keep, then remove the markers before committing.

  • <<<<<<< HEAD — your current branch's version
  • ======= — separator between the two versions
  • >>>>>>> bob — the incoming branch's version
  • Keep both, keep one, or write something entirely new — it's your call
team.txt — conflicted
Team members: Alice, Bob <<<<<<< HEAD Alice is working on the login feature. ======= Bob is working on the payment gateway. >>>>>>> bob # After editing — keep both lines, remove markers: Team members: Alice, Bob Alice is working on the login feature. Bob is working on the payment gateway.
§5 — resolving conflicts

Conflict Resolution Playbook

  1. 1
    Identify conflicted files git status
  2. 2
    Open each file in your editor (VS Code, nano) code .   or   nano team.txt
  3. 3
    Edit to desired content — remove ALL markers <<<<<<<   =======   >>>>>>> must go
  4. 4
    Stage the resolved file git add team.txt
  5. 5
    Commit the merge git commit -m "Merge bob — resolve conflict"
  6. 6
    Test your code! git log --oneline --graph --all
VS Code Tip
VS Code shows conflict buttons above markers: Accept Current Change (HEAD) Accept Incoming Change (other branch) Accept Both Changes (keep both) Compare Changes (diff view)
Abort if needed
$ git merge --abort # undo the merge entirely # Returns to pre-merge state
§6 — github & remotes

From Local to Remote — GitHub

A remote is a copy of your repo hosted online. GitHub is the most popular host. It provides backup, collaboration, and your public portfolio.

  • Backup — your code survives a laptop crashPush often = more safety net
  • Collaboration — teammates clone and contributeEveryone works from the same source of truth
  • Portfolio — employers look at your GitHub profileGreen squares = proof you ship code
Connect to GitHub
# 1. Create repo on GitHub (no README init) # 2. Link your local repo to remote: $ git remote add origin https://github.com/you/repo.git # 3. Push for the first time: $ git push -u origin main Branch 'main' set up to track remote branch 'main' from 'origin'. # After the first push — just use: $ git push # push $ git pull # get latest from remote # Using HTTPS? Use a Personal Access Token # as your password (not your GitHub password)

§7 — hands-on practice

Mini Project — Build a Team Page

Work in pairs. One repo, two contributors, real merge conflict.

  1. 1
    Pair up — decide who creates the repo (Owner)
  2. 2
    Owner: init repo, commit HTML template, push to GitHub
  3. 3
    Both: create branches — feature-header / feature-footer
  4. 4
    Each partner edits the same line to force a conflict
  5. 5
    Merge one branch (OK), then the other (CONFLICT!)
  6. 6
    Resolve together, push, and open PRs on GitHub
Repo Owner
  • Creates repo on GitHub
  • Makes first commit (HTML template)
  • Adds contributor as collaborator
  • Handles the merge (resolves conflict)
Contributor
  • Clones the owner's repo
  • Creates feature branch and commits
  • Pushes branch, opens a PR
  • Reviews and comments on owner's PR

§8 — troubleshooting

Common Questions & Fixes

QWhat if I committed on the wrong branch? AUse git cherry-pick <hash> to copy the commit to the right branch, then git reset HEAD~1 to undo it on the wrong one. (Advanced — ask instructor.)
QHow do I undo a merge that's gone wrong? AIf the merge hasn't been committed yet: git merge --abort. If it's committed: git revert -m 1 HEAD (creates an undo commit).
QWhy did I get a "detached HEAD" state? AYou checked out a commit hash directly instead of a branch. Create a branch to save your work: git checkout -b rescue-branch, then continue normally.
QPermission denied (publickey) when pushing? AEither use HTTPS with a Personal Access Token (not your password), or set up SSH keys: ssh-keygen then add the public key to GitHub Settings → SSH Keys.
